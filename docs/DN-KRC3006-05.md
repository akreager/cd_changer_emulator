## 1. Purpose & Scope

This document defines the architecture, technology stack, and design rationale for the web application component of the CD changer emulator. The web app runs on the Raspberry Pi 5 and provides a browser-based interface for managing the music library, creating playlists, and loading virtual magazines. It does **not** control playback — all playback commands originate exclusively from the KRC-3006 head unit.

### 1.1 What the Web App Does

- **Track library management** — Browse, search, and organize a FLAC music library with an iTunes-like interface (artist, album, genre views).
- **Playlist creation** — Create, edit, save, and delete playlists from the track library.
- **Virtual magazine loading** — Assign playlists to virtual disc slots in a magazine. Save and swap magazines.
- **Library scanning** — Automatically detect and ingest new music added to the NVMe SSD.
- **System status** — Display current playback state, disc/track numbers, battery voltage, WiFi mode, and power state (read-only; informational, not interactive).

### 1.2 What the Web App Does Not Do

- **No playback controls.** No play, pause, skip, or seek buttons in the UI. The head unit is the sole playback controller.
- **No filesystem-level disc management.** Users never manually move files into numbered folders. The database and magazine abstraction handle all mapping.
- **No multi-format support.** The library ingests FLAC files only. CUE+BIN or other image formats must be converted to individual FLAC files externally before import (see Section 8).

### 1.3 Primary Use Context

The primary interaction device is a **smartphone or tablet**. The UI must be mobile-first and touch-friendly. Desktop use is secondary (bulk library management from a laptop at home).

### 1.4 Target Hardware

- Raspberry Pi 5 with 1 TB NVMe SSD
- Thermal management via DVFS; optional underclocking in `/boot/firmware/config.txt` (`arm_freq=1800`, `arm_freq_min=500`) for automotive thermal margin
- Active cooler sufficient for this workload (audio decoding + web serving barely loads the A76 cores)

## 2. Head Unit Command Set & Display Constraints

The KRC-3006 is a shaft-style cassette receiver. Its CD changer controls and display capabilities constrain what the emulator can present to the user through the head unit.

### 2.1 Available Head Unit Controls (CD Changer Mode)

Source: KRC-3006 Owner's Manual (B54-3272-00)

| Action | Physical Control | Protocol Command |
|---|---|---|
| Play / Stop | Button ④ (briefly press) | Start/pause playback |
| Next track | Knob ⑦ twist CW once | Skip to next track |
| Previous track / restart | Knob ⑦ twist CCW once | Skip to start of current or previous track |
| Fast forward (audible) | Knob ⑦ twist CW and hold | Scrub forward in current track |
| Rewind (audible) | Knob ⑦ twist CCW and hold | Scrub backward in current track |
| Next disc | Knob ⑦ press | Advance to next virtual disc (playlist) |
| Track repeat / Disc repeat / Off | Button ② (cycles) | Loop current track, loop current disc, or normal play |
| Track scan | Button ⑥ | Play first 10 seconds of each track on current disc |
| Disc scan | Button ① | Play first 10 seconds of track 1 on each disc in magazine |
| Random play | Button ⑥ | Shuffle tracks on current disc |
| Reset | Button ③ | Reinitialize the receiver |

**Notable limitation:** There is no direct disc number selection. The user can only advance to the *next* disc sequentially via the knob press. This is significant for the magazine overflow design (Section 5) — wrapping from disc 10 back to disc 1 will feel natural since the user is already accustomed to sequential advancement.

### 2.2 Display Capabilities

In CD changer mode, button ③ toggles the display between **Track Number** and **Clock**. The display shows:

- Track number (e.g., `T-01`)
- Disc number indicator (1–6 shown on the segmented display; how 7–10 are displayed is TBD from protocol capture)
- Playback state indicators (play icon, repeat icon, random icon)

**The head unit cannot display elapsed track time.** The display has no time field in CD changer mode — only track number and disc number are shown. However, the playback daemon will track elapsed time internally via python-mpv's `time-pos` callback for two reasons:

1. The Kenwood protocol status frames may include time data even if this head unit cannot display it (protocol capture will confirm).
2. A future supplemental display (e.g., OLED driven by ATtiny I2C) or the web UI's now-playing status panel can use it.

## 3. Recommended Technology Stack

### 3.1 Stack Summary

| Layer | Choice | Rationale |
|---|---|---|
| Web framework | **FastAPI + Uvicorn** | Native async, WebSocket, auto-generated API docs, SQLModel integration |
| Database | **SQLite (WAL mode + FTS5)** | Zero-config, 1–5 MB RAM, sub-ms queries, concurrent read/write via WAL |
| ORM | **SQLAlchemy 2.0 + Alembic** | Type-safe `Mapped[]` columns, native async, batch-mode migrations for SQLite |
| Frontend | **HTMX + Alpine.js + DaisyUI/Tailwind CSS** | No build tools, ~30 KB JS total, mobile-first, touch-friendly |
| Metadata | **TinyTag** (read) | Lightweight, FLAC-native, fast bulk scanning |
| IPC | **ZeroMQ (PUB/SUB + PUSH/PULL)** | ~33 µs latency, no broker process, auto-reconnect |
| Audio engine | **python-mpv (libmpv)** | Gapless playback, 7 MB RAM, I2S DAC output, event callbacks |
| File watching | **watchdog (inotify)** | Kernel-level change detection, no polling |

### 3.2 Why FastAPI Over Flask

DN-KRC3006-01 originally specified Flask. FastAPI is recommended instead for three reasons specific to this project:

1. **Native WebSocket support.** Pushing real-time playback status (disc, track, state) to the browser requires WebSockets. FastAPI inherits first-class WebSocket handling from Starlette. Flask requires the Flask-SocketIO extension with greenlet concurrency — a workaround, not native support.

2. **Async architecture on constrained hardware.** FastAPI runs on ASGI (Uvicorn), handling REST requests, WebSocket connections, and background tasks (library scanning) on a single event loop. In benchmarks, Flask deadlocks at ~400 rps under mixed workloads while FastAPI remains stable at ~9,000 rps. On a Pi, this translates to responsiveness during library scans.

3. **Auto-generated API documentation.** The interactive `/docs` endpoint (Swagger UI) lets every API endpoint be tested in the browser with auto-generated forms. This is invaluable for a developer new to web app creation — no need to write curl commands or use Postman during development. Combined with Pydantic validation, FastAPI catches type errors at development time rather than runtime.

**Note:** DN-KRC3006-01 Section 4.7 references "Flask" in the web interface description and the packages list. When implementation begins, those references should be updated to reflect FastAPI + Uvicorn. The UART communication, WiFi management, and power sequencing in DN-KRC3006-01 remain unchanged.

### 3.3 Why SQLite

SQLite with WAL mode is purpose-built for this workload: a single-user, embedded-device music library.

- **Zero configuration.** No daemon, no passwords, no ports, no service files. A single `import sqlite3` and the database exists.
- **Minimal resources.** 1–5 MB RAM as an embedded library versus 50–100 MB for PostgreSQL's server process.
- **WAL mode solves concurrency.** The web UI performs frequent reads (browsing, searching) while the playback daemon performs occasional writes (updating last-played timestamp). WAL ensures readers never block writers:
  ```sql
  PRAGMA journal_mode=WAL;
  PRAGMA busy_timeout=5000;
  ```
- **FTS5 for library search.** Full-text search with BM25 ranking, prefix queries, and column weighting — sub-millisecond search across 50,000+ tracks.
- **Backup is trivial.** `VACUUM INTO 'backup.db'` creates a consistent snapshot, or just copy the file. Litestream can continuously stream WAL changes for automated backup with near-zero overhead.
- **Proven at scale for music.** Navidrome and beets both use SQLite exclusively for libraries of 100k+ tracks.

### 3.4 Why HTMX + Alpine.js (Not React/Vue)

The deployment story is decisive. HTMX requires **no Node.js, no webpack, no Vite, no build pipeline** — not on the Pi, not on the dev machine. It is a 14 KB JavaScript file included via a `<script>` tag. Alpine.js adds 15 KB for client-side reactivity. The entire interactive frontend weighs ~30 KB. Compare this to a React + Material UI SPA that ships 200–500 KB and requires a Node.js toolchain to build.

HTMX works by issuing HTTP requests from HTML attributes and swapping server-rendered HTML fragments into the DOM. This maps naturally to FastAPI + Jinja2 template rendering — every user interaction is just another server endpoint returning an HTML fragment.

**Key interaction patterns:**

- **Library search:** `hx-trigger="keyup changed delay:300ms"` provides live search with automatic 300ms debounce.
- **Drag-and-drop playlist reordering:** SortableJS integration via HTMX's official Sortable example. Dragging triggers `hx-trigger="end"` which posts the new order to the server. Alpine.js `x-sort` plugin provides touch-compatible drag-and-drop.
- **Real-time playback status:** HTMX WebSocket extension (`hx-ext="ws"`, `ws-connect="/ws/playback"`) receives HTML fragments from the server and swaps them into the DOM via out-of-band updates.
- **Infinite scroll for large libraries:** `hx-trigger="revealed"` on a sentinel element triggers pagination without a "Load More" button.

**DaisyUI + Tailwind CSS** provides the mobile-first design system. DaisyUI layers semantic component classes (`btn`, `card`, `drawer`, `navbar`, `progress`) on Tailwind, offering 30+ themes including dark mode — useful for nighttime driving. Minimum 44×44 px touch targets ensure mobile usability.

**Upgrade path:** If HTMX eventually feels limiting (complex drag choreography, offline PWA), **Svelte** is the recommended SPA alternative. It compiles to vanilla JS with zero runtime (50–150 KB bundles). Build on the dev machine, rsync static files to the Pi. Node.js never touches the Pi.

## 4. Data Model

### 4.1 Core Tables

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────┐
│   tracks     │     │  playlist_tracks  │     │  playlists   │
├──────────────┤     ├──────────────────┤     ├──────────────┤
│ id (PK)      │──┐  │ id (PK)          │  ┌──│ id (PK)      │
│ title        │  └─→│ track_id (FK)    │  │  │ name         │
│ artist       │     │ playlist_id (FK) │←─┘  │ description  │
│ album        │     │ position (int)   │     │ created_at   │
│ album_artist │     └──────────────────┘     │ updated_at   │
│ genre        │                               └──────────────┘
│ track_number │
│ disc_number  │     ┌──────────────────┐     ┌──────────────┐
│ year         │     │  magazine_slots   │     │  magazines   │
│ duration     │     ├──────────────────┤     ├──────────────┤
│ file_path    │     │ id (PK)          │  ┌──│ id (PK)      │
│ file_size    │     │ magazine_id (FK) │←─┘  │ name         │
│ file_mtime   │     │ slot_number (int)│     │ is_active    │
│ sample_rate  │     │ playlist_id (FK) │     │ created_at   │
│ bit_depth    │     │ UNIQUE(mag,slot) │     │ updated_at   │
│ album_art    │     └──────────────────┘     └──────────────┘
│ metadata_json│
│ created_at   │
│ updated_at   │
└──────────────┘
```

### 4.2 Key Constraints

- `playlist_tracks.position` — Integer column for explicit track ordering within a playlist. Reordering updates positions via a batch UPDATE.
- `magazine_slots` — `UNIQUE(magazine_id, slot_number)` enforces that each slot in a magazine maps to exactly one playlist.
- `magazine_slots.slot_number` — No hard cap at the database level (see Section 5, Magazine Overflow). The 10-slot presentation limit is enforced in the playback daemon, not the schema.
- `magazines.is_active` — Boolean flag; exactly one magazine is active at a time (enforced in application logic).
- `tracks.metadata_json` — JSON column (via SQLite `json1`) for extensible metadata: ReplayGain values, custom tags, import source, etc.
- `tracks.file_mtime` — Stores the file's modification timestamp at scan time. Enables incremental rescans by comparing against `os.stat()` without re-reading metadata.

### 4.3 FTS5 Virtual Table

```sql
CREATE VIRTUAL TABLE tracks_fts USING fts5(
    title, artist, album, album_artist, genre,
    content='tracks',
    content_rowid='id',
    tokenize='porter unicode61'
);
```

Synchronized via triggers on INSERT, UPDATE, DELETE against the `tracks` table. Supports ranked search with column weighting (boost title and artist over genre).

## 5. Virtual Magazine Design

### 5.1 Core Concept

A magazine is an ordered collection of playlists assigned to numbered slots. The emulator presents slots 1–10 to the head unit as disc numbers 1–10 via the `MAG:bbbbbbbbbb` bitmap in the ATtiny UART protocol (see DN-KRC3006-01, Section 4.8). Only one magazine is active at a time.

### 5.2 Magazine Overflow ("Playlist of Playlists")

**Use case:** An artist has more than 10 albums. The user wants to listen to their full discography in chronological order across a single magazine.

**Design:** A magazine may contain more than 10 playlists. The first 10 are presented to the head unit as discs 1–10. When the last track of the 10th disc finishes, the playback daemon:

1. Sends `MUTE_ON` to the ATtiny.
2. Internally advances to playlist 11 and remaps the head unit's disc 1–10 window to playlists 11–20 (or 11–N for the final partial bank).
3. Updates the `MAG:` bitmap for the new bank (some slots may be empty if fewer than 10 playlists remain).
4. Sends `DISC:01` and `TRACK:01` to the ATtiny.
5. Sends `MUTE_OFF` after audio is ready.
6. Resumes playback from playlist 11, track 1.

From the head unit's perspective, this looks like a magazine swap — the disc number jumps from 10 back to 1. Since the head unit only supports sequential next-disc advancement (knob ⑦ press), this wrapping behavior is transparent and natural.

**Configuration:** This overflow behavior is a per-magazine toggle in the web UI:

- **Overflow ON (default for discography magazines):** Disc 10 rolls over to the next bank. Playback continues seamlessly.
- **Overflow OFF (strict 10-disc mode):** Playback stops at the end of disc 10, mimicking a real 10-disc changer.

### 5.3 Magazine Resolution Path

When the ATtiny relays a head unit command like `CMD:DISC:03 TRACK:05`, the playback daemon resolves it:

```
Active magazine
  → current bank offset (0, 10, 20...)
  → bank_offset + 3 = playlist index 3 (or 13, 23...)
  → playlist at that index
  → track at position 5 within the playlist
  → file_path from tracks table
  → python-mpv begins playback
```

This resolution must complete in under 1 ms. The daemon pre-loads the active magazine's playlists into an in-memory cache on startup and on magazine change. The cached data is small — 10 playlists × 100 tracks × ~200 bytes ≈ 200 KB.

### 5.4 Cache Invalidation

When the web UI modifies a playlist that is in the active magazine:

1. The FastAPI endpoint writes the change to SQLite.
2. It publishes an invalidation message via ZeroMQ: `{"event": "playlist_changed", "playlist_id": 42}`.
3. The playback daemon receives the message, reloads the affected playlist from SQLite into its in-memory cache.
4. If the changed playlist is the currently playing disc, playback continues from the current position (the daemon does not restart or skip).

## 6. Two-Process Architecture

The system is split into two Python processes that communicate via ZeroMQ and share a SQLite database file.

### 6.1 Process 1: Web Application (FastAPI)

**Responsibilities:**
- Serve the web UI (Jinja2 templates + HTMX responses)
- REST API for library, playlist, and magazine CRUD
- Library scanning (background task)
- WebSocket endpoint for pushing playback status to browsers
- Receive playback state from daemon via ZeroMQ SUB, broadcast to WebSocket clients

**Does not:** Decode audio, communicate with the ATtiny, or manage the Kenwood protocol.

### 6.2 Process 2: Playback Daemon

**Responsibilities:**
- UART communication with the ATtiny (pyserial, 115200 baud, 8N1)
- Parse Kenwood protocol commands relayed by ATtiny (`CMD:xx xx xx\n`)
- Resolve disc/track numbers through the magazine cache to file paths
- Control python-mpv for audio playback (gapless, I2S DAC output)
- Publish playback state via ZeroMQ PUB (JSON: disc, track, elapsed time, state)
- Receive commands from web app via ZeroMQ PULL (magazine changes, cache invalidation)

**Does not:** Serve HTTP, render HTML, or interact with browsers directly.

### 6.3 ZeroMQ IPC

Two socket pairs over Unix IPC transport:

| Channel | Pattern | Transport | Purpose |
|---|---|---|---|
| State broadcast | PUB/SUB | `ipc:///tmp/cdchanger-state.sock` | Daemon publishes playback state; web app subscribes |
| Commands | PUSH/PULL | `ipc:///tmp/cdchanger-cmd.sock` | Web app pushes commands; daemon pulls |

**State message format (JSON, ~1/second during playback):**
```json
{
  "state": "playing",
  "disc": 3,
  "track": 5,
  "title": "Take Five",
  "artist": "Dave Brubeck",
  "elapsed": 47.2,
  "duration": 324.0,
  "magazine": "Jazz Classics",
  "bank": 0
}
```

**Command message format (JSON, event-driven):**
```json
{"command": "set_magazine", "magazine_id": 5}
{"command": "invalidate_playlist", "playlist_id": 42}
```

**Why ZeroMQ over alternatives:**
- **No broker.** Unlike Redis or MQTT (Mosquitto), there is no third process to install and manage.
- **~33 µs latency** for messages under 4 KB — imperceptible for button presses.
- **Auto-reconnect.** If either process restarts, ZeroMQ re-establishes the connection.
- **pyzmq** is mature, well-documented, and pip-installable.

### 6.4 Graceful Degradation

- If the web app crashes, head unit control continues uninterrupted through the daemon.
- If the head unit is disconnected, the web app still functions for library/playlist management.
- The daemon writes its state to a JSON file periodically so it can resume playback position after a restart.

## 7. Audio Metadata & Library Scanning

### 7.1 Metadata Library: TinyTag

Since the library is FLAC-only, metadata reading is straightforward. **TinyTag** (MIT license) is the recommended library for the read-only scanning pass:

```python
from tinytag import TinyTag
tag = TinyTag.get('/music/album/01_track.flac')
# tag.title, tag.artist, tag.album, tag.duration, tag.track, ...
```

TinyTag reads only the minimum bytes from each file header, never loading full files into memory. Its v2.0 release added structured album art access (`tag.images.front_cover`) with MIME type and dimensions.

If tag *writing* is ever needed (e.g., correcting metadata from the web UI), **Mutagen** is the appropriate library for that operation. TinyTag is read-only.

### 7.2 Scanning Strategy

The scanner follows Navidrome's proven approach, adapted for Python:

**Initial bulk scan:**
1. Walk the music directory tree with `os.walk()`.
2. Filter for `.flac` extensions.
3. Read metadata with TinyTag for each file.
4. Batch-insert into SQLite every 100–500 files within a single transaction.
5. WAL mode allows the web UI to browse already-scanned tracks while scanning continues.

**Incremental updates:**
1. Store each file's `mtime` and `file_size` in the `tracks` table.
2. On rescan, compare `os.stat()` results against stored values.
3. Only re-read metadata for new or modified files. Remove database entries for deleted files.

**Filesystem watching (real-time):**
1. Python's `watchdog` library uses Linux inotify (kernel-level, no polling).
2. Debounce window of 10 seconds before processing — file copy operations generate multiple rapid events.
3. On Pi, the default inotify watch limit may need increasing: `sudo sysctl fs.inotify.max_user_watches=524288`.

**Background execution:**
Scanning runs as a FastAPI background task, never blocking the web UI. Expose a `/api/scan/start` endpoint (POST) and push progress updates via WebSocket.

### 7.3 Album Art

1. Extract embedded art during the scan pass.
2. Save to the filesystem as `{album_hash}.jpg`.
3. Generate thumbnails lazily on first request using **Pillow** (64×64 for grid views, 300×300 for album detail).
4. Serve with aggressive HTTP cache headers (`Cache-Control: max-age=31536000, immutable`).
5. Fallback chain for missing art: embedded tag → `cover.jpg` / `folder.jpg` in album directory → placeholder SVG.

## 8. CUE Sheet Handling

CUE+BIN (raw CD image) and CUE+FLAC (single-image FLAC with cue sheet) are **not natively supported** by the emulator's library scanner or playback engine. These formats must be converted to individual FLAC files before import.

**Rationale:** The playback daemon needs per-track file paths, the database needs per-track metadata, and the head unit needs accurate per-track numbers and durations. Seeking within a monolithic image file by parsing CUE timestamps at runtime adds fragile complexity to every layer.

**Recommended external tool:** FFcuesplitter (`pip install ffcuesplitter`) can split CUE+FLAC images into individual FLAC files with embedded Vorbis comment tags (title, artist, album, track number) extracted from the CUE sheet. It can also convert CUE+BIN (raw PCM) to FLAC in the same pass via FFmpeg.

**Future consideration:** A "CUE import" helper could be added to the web UI that accepts a CUE+FLAC upload, runs FFcuesplitter in the background, and ingests the resulting individual FLAC files. This is a convenience feature, not a core requirement.

## 9. Playback Engine: python-mpv

**python-mpv** wraps libmpv via ctypes and provides the best balance of features for this project:

- **Gapless audio** via `--gapless-audio=yes` with prefetch buffering (`--prefetch-playlist=yes`).
- **I2S DAC output** through ALSA: `--audio-device=alsa/default`.
- **Resource usage:** ~7 MB RAM, <10% CPU on Pi 5 for FLAC decoding.
- **Event callbacks** for playback state tracking:
  - `@player.property_observer('time-pos')` — Elapsed time (drives the 1/second state broadcast).
  - `@player.event_callback('end-file')` — Track completion (triggers next-track logic or disc advance).
- **Programmatic control:** `player.playlist_pos`, `player.seek()`, `player.pause`, `player.playlist_next()` — all needed for responding to head unit commands.

**Alternatives considered:**
- **MPD (Music Player Daemon):** Battle-tested, native gapless, decades of optimization. But adds a third process to manage. Considered if python-mpv proves unreliable.
- **VLC (python-vlc):** Lacks gapless playback until VLC 4.0. Not suitable.

## 10. Web UI Pages

The web application consists of the following views, all mobile-first and served via Jinja2 templates with HTMX interactivity.

### 10.1 Track Library

The primary library browsing interface, modeled after iTunes' column browser pattern.

**Views:**
- **Artists** — Alphabetical list with album counts. Tap to filter.
- **Albums** — Grid of album art thumbnails with artist/title. Tap for track list.
- **All Tracks** — Sortable table: title, artist, album, duration, genre, year. Paginated (100 per page, infinite scroll via HTMX `hx-trigger="revealed"`).
- **Search** — Live search bar with 300ms debounce, powered by FTS5. Results update in-place without page reload.

**Actions:**
- Select tracks → "Add to playlist" (dropdown of existing playlists or "New playlist").
- Select album → "Add album to playlist" (adds all tracks in album track-number order).
- Bulk select for batch operations.

### 10.2 Playlist Manager

**List view:** All saved playlists with name, track count, total duration.

**Detail view (single playlist):**
- Ordered track list with drag-and-drop reordering (SortableJS + HTMX).
- Remove tracks (swipe or delete button).
- Add tracks (search inline or link to library view).
- Rename, duplicate, delete playlist.
- Total duration display (sum of track durations).

### 10.3 Magazine Manager

**List view:** All saved magazines with name, slot count, active indicator.

**Detail view (single magazine):**
- 10 numbered slot cards (or more, if overflow is enabled).
- Each slot shows: assigned playlist name, track count, duration. Or "Empty" with a dashed border.
- Drag-and-drop playlists onto slots.
- "Add slot" button (for overflow magazines beyond 10).
- Toggle: "Continue past disc 10" (overflow on/off).
- "Load this magazine" button → sets as active, pushes update to playback daemon via ZeroMQ.

### 10.4 Now Playing (Read-Only Status)

A persistent status bar or dedicated page showing current playback state received via WebSocket:

- Current magazine name and bank indicator (if overflow active).
- Disc number / total discs in current bank.
- Track number / total tracks on current disc.
- Track title, artist, album.
- Elapsed time / total duration (even though the head unit cannot display this).
- Playback state icon (playing, stopped, scanning, random, repeat mode).
- Album art.

**This page has no interactive controls.** It is purely informational.

### 10.5 System Dashboard

- Battery voltage (from ATtiny `VBATT` reports).
- WiFi mode (AP/Client) and connected clients.
- Power state (RUNNING, STANDBY, UPDATE).
- SSD usage (used / total).
- Library statistics (track count, total duration, last scan time).
- "Rescan library" button.
- Links to WiFi settings, power settings, firmware update (carried forward from DN-KRC3006-01 Section 4.7).

## 11. Existing Projects for Reference

These open-source projects informed this design. None implements the virtual disc-slot mapping unique to this project, but each contributes architectural patterns worth studying.

### 11.1 Navidrome

- **Repo:** github.com/navidrome/navidrome (~20k stars)
- **Stack:** Go + SQLite + React + Material UI
- **Relevance:** Closest overall reference architecture. Runs on Pi Zero. Uses SQLite with WAL mode. 4-phase scanning pipeline (folder scan → missing track detection → album refresh → playlist import). Filesystem watching with 10-second debounce. Its multi-library feature is the closest existing concept to virtual disc slots.
- **Study for:** Scanning pipeline, SQLite WAL patterns, library data model.

### 11.2 Koel

- **Repo:** github.com/koel/koel (~17k stars)
- **Stack:** PHP/Laravel + Vue.js
- **Relevance:** Best-in-class iTunes-like web UI. Sidebar navigation, album grids, drag-and-drop playlist management. Too heavy for Pi deployment (PHP + MySQL), but the UI design is directly applicable.
- **Study for:** UI layout and interaction patterns.

### 11.3 Mopidy

- **Repo:** github.com/mopidy/mopidy
- **Stack:** Python (Pykka actor model)
- **Relevance:** Most relevant Python architecture. Actor-based separation of frontends (HTTP, MPD) from backends (local, Spotify) through a core broker. This maps to our project: head unit serial and web UI are two frontends, magazine resolution is the core, local file playback is the backend. Has proven Pi GPIO support.
- **Study for:** Frontend/core/backend separation, Python architecture.
- **Web client:** Iris (github.com/jaedb/Iris) — polished React/Redux UI.

### 11.4 Volumio

- **Repo:** github.com/volumio/Volumio2
- **Stack:** Node.js + Socket.IO + Angular
- **Relevance:** Socket.IO-first API where every interaction flows through WebSocket events. Dedicated playback state machine module. Good model for real-time state synchronization between multiple interfaces.
- **Study for:** WebSocket event architecture, playback state machine pattern.
